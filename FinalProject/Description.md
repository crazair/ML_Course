# RepoClassifier: ML-утилита для автоматического маппинга GitLab-репозиториев на продукты

## 1. Постановка задачи

### Контекст

В компании количество репозиториев в корпоративном GitLab растёт непрерывно: создаются новые сервисы, библиотеки, форки, инфраструктурные конфигурации. Параллельно развивается продуктовый ландшафт — появляются новые информационные системы (продукты). Определить, к какому именно продукту относится конкретный репозиторий, становится нетривиальной задачей, особенно когда маппинг ведётся вручную и быстро устаревает.

### Формулировка задачи

Разработать AI-утилиту на Python, которая на основе набора признаков автоматически определяет принадлежность репозитория в корпоративном GitLab к продукту. Задача формулируется как **задача мультиклассовой классификации**: на входе — набор признаков репозитория, на выходе — название продукта (или top-N наиболее вероятных продуктов с оценкой уверенности).

### Название проекта

**RepoClassifier** — интеллектуальный классификатор принадлежности репозиториев к продуктам.

---

## 2. Бизнес-ценность

Точное определение привязки репозитория к продукту необходимо для решения следующих практических задач:

- **Аллокация дефектов** — корректное распределение багов и инцидентов по продуктовым командам.
- **Привязка результатов сканирования на уязвимости** — отчёты SAST/DAST/SCA должны агрегироваться на уровне продукта для его оценки.
- **Определение покрытия юнит-тестами** — метрику code coverage имеет смысл видеть в том числе в разрезе продукта, а не только в виде отдельного репозитория.
- **Сокращение ручного труда** — устранение необходимости регулярного ручного аудита маппинга, что экономит рабочее время инженеров.
- **Онбординг новых репозиториев** — автоматическое предложение продукта при создании нового репозитория.

## 3. Критерии (признаки) для классификации

### 3.1 Справочник маппинга «репозиторий → продукт»

Исторический справочник, в котором уже зафиксированы известные связи. Используется как **обучающая выборка** и как один из входных сигналов (для частично размеченных кейсов).

### 3.2 Количество и доля коммитов от сотрудников продукта

На основе справочника «сотрудник → продукт» рассчитывается, какая доля коммитов в репозиторий приходится на сотрудников каждого продукта. Это один из самых сильных сигналов — люди чаще всего коммитят в репозитории своего продукта.

**Производные признаки:**

- Абсолютное число коммитов по каждому продукту.
- Доля коммитов top-1 продукта от общего числа.
- Энтропия распределения коммитов по продуктам (низкая энтропия = сильная принадлежность).
- Динамика: доля коммитов за последние 30/90/180 дней (приоритет свежим данным).

### 3.3 Структура namespace/group и именование репозитория

GitLab-группы и подгруппы часто отражают организационную или продуктовую структуру. Название самого репозитория нередко содержит префикс или суффикс продукта.

**Производные признаки:**

- Полный путь репозитория (group/subgroup/repo) — токенизированный и закодированный.
- Совпадение токенов пути с названиями известных продуктов (fuzzy matching score).
- Уровень вложенности и принадлежность группы к известному продуктовому namespace.

### 3.4 Анализ текстового содержимого репозитория

Файлы `README.md`, `CODEOWNERS`, конфигурации CI/CD (`.gitlab-ci.yml`), манифесты зависимостей (`pom.xml`, `package.json`, `requirements.txt`, `go.mod`) и переменные окружения содержат ценную информацию.

**Производные признаки:**

- TF-IDF или эмбеддинги текста из README (упоминания продуктовых названий, доменных терминов).
- Список зависимостей — пересечение с зависимостями известных репозиториев продукта.
- Содержимое файла `CODEOWNERS` — пересечение с сотрудниками продуктов.

### 3.5 Метаданные и активность репозитория

GitLab API предоставляет богатый набор метаданных, которые косвенно сигнализируют о принадлежности.

**Производные признаки:**

- Topics/tags репозитория.
- Связанные issues и merge requests — авторы, assignees, labels.
- Частота и паттерны активности (есть ли корреляция с релизными циклами конкретного продукта).
- Fork-связи: если репозиторий является форком — от какого родительского репозитория, и к какому продукту тот относится.

## 4. Данные для обучения

### Источники данных

| Источник | Описание | Способ получения              |
|---|---|-------------------------------|
| GitLab API | Метаданные репозиториев, коммиты, MR, CI/CD | GitLab REST API               |
| Справочник «репозиторий → продукт» | Размеченные примеры (ground truth) | Справочник в Confluece        |
| Справочник «сотрудник → продукт» | Принадлежность разработчиков к командам | Справочник оргструктуры       |
| Содержимое репозиториев | README, конфиги, манифесты | GitLab REST API   |

### Формирование датасета

1. Через GitLab API собрать полный список репозиториев с метаданными.
2. Для каждого репозитория получить историю коммитов с email/username авторов.
3. Через справочник сотрудников рассчитать признаки п. 3.2.
4. Извлечь текстовые признаки (README, CI/CD, зависимости).
5. Разметить датасет с помощью справочника маппинга.
6. Репозитории без разметки — кандидаты для предсказания (inference).

### Разбиение данных

- **Train** — 80% размеченных репозиториев.
- **Test** — 20% для финальной оценки.
- Стратификация по продуктам для сохранения баланса классов.

## 5. Подход к решению

### Почему ML, а не правила?

Эвристический подход (например, «если >80% коммитов от команды продукта X → репозиторий принадлежит X») работает для простых случаев, но плохо масштабируется: пороги приходится подбирать вручную, новые критерии сложно комбинировать, пограничные случаи (shared-библиотеки, кросс-командные сервисы) не покрываются. ML-модель способна взвесить все признаки и обучиться на исторических данных.

### Про модели

| # | Модель | Обоснование                                                                                                                                                                |
|---|---|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1 | **CatBoost** (основная) | Нативная работа с категориальными признаками (namespace, topics), устойчивость к пропускам данных, высокое качество на табличных задачах «из коробки». Поддержка text-фичей. |
| 2 | **Fine-tuned LLM** (опционально) | Дообученная модель компании для формирования эмбеддингов из текстовых полей (README, path).                            |

### Архитектура решения

```
┌──────────────┐     ┌──────────────────┐     ┌───────────────┐
│  GitLab API  │────▶│  Feature Engine  │────▶│  ML Pipeline  │
└──────────────┘     │  (сбор и расчёт  │     │  CatBoost     │
┌──────────────┐     │   признаков)     │     └───────┬───────┘
│  Справочники │────▶│                  │             │
└──────────────┘     └──────────────────┘             ▼  
                                               ┌────────────────┐
                                               │  Предсказание  │
                                               │  top-N продукт │
                                               │  + confidence  │
                                               └────────────────┘
```

## 6. Метрики качества

### Технические метрики (на валидации)

| Метрика | Описание | Обоснование |
|---|---|---|
| **Accuracy** | Доля верных предсказаний | Базовая метрика, понятна всем |
| **Macro F1-score** | Среднее F1 по всем классам | Учитывает дисбаланс классов (есть продукты с 2-3 репозиториями и продукты с сотнями) |

### Целевые значения

- Accuracy ≥ 95% на тестовой выборке.
